# Feature status - full pipeline view and WIKI_REPORT refresh

## Usage
/feature-status

## What this command does

1. Read `knowledge/wiki/SCHEMA.md`
2. Read `knowledge/wiki/SETTINGS.md` if present
3. Read `knowledge/wiki/index.md`
4. Read all feature files in `knowledge/wiki/features/`
5. Read the linked design, platform-requirements, api-contract, advisory-review, and
   business-rule files needed to compute:
   - feature counts by lifecycle stage
   - open questions grouped by owner
   - canonical blocker-category counts
   - recently updated wiki pages
   - suggested next actions
6. Output the current pipeline view for the user
7. Write or refresh `knowledge/wiki/WIKI_REPORT.md`

## WIKI_REPORT rules

- `feature-status` is the only command that writes `knowledge/wiki/WIKI_REPORT.md`
- `WIKI_REPORT.md` is generated and read-only
- `WIKI_REPORT.md` is an orientation artifact, not the source of truth
- if `WIKI_REPORT.md` disagrees with the underlying wiki files, the underlying wiki files win
- `WIKI_REPORT.md` must stay short enough to read before drilling into the full wiki
- include a short pointer telling the user to run `/lint-wiki` for structural issues
- do not put full lint conclusions into `WIKI_REPORT.md`

If `knowledge/wiki/SETTINGS.md` is absent or invalid, fall back to
`wiki-stale-after-days: 14` and tell the user.

## Notes for the agent

- Do not modify feature files, index.md, or log.md in this command.
- Only write `knowledge/wiki/WIKI_REPORT.md`.
- Use the canonical blocker vocabulary from `knowledge/wiki/SCHEMA.md`.
- If the report cannot be created, still return the pipeline view and clearly say the
  report refresh failed.
